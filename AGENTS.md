# AI海龟汤游戏开发指令

## 项目概述
使用 React + TypeScript + Tailwind CSS 开发一个 AI 海龟汤游戏网站。产品核心玩法为：玩家阅读“汤面”后，通过输入是非类问题与 AI 主持人交互，AI 仅根据预设汤底返回“是 / 否 / 无关”或在特定场景下提供分级提示，帮助玩家逐步还原真相。

## 开发规范
- 使用 TypeScript，确保类型安全，禁止大面积使用 `any`
- 使用函数式组件 + React Hooks 开发
- 使用 Tailwind CSS 编写样式，优先复用现有样式片段
- 组件职责单一，尽量拆分为可复用组件
- 核心逻辑与 UI 展示分层，避免把业务逻辑堆在页面组件中
- 所有 AI 调用必须经过服务端中转，不允许前端直连模型 API
- 关键逻辑必须添加注释，尤其是 AI 判定、状态流转、提示系统、通关判定等部分
- 优先保证 MVP 可用，再逐步增强氛围、动画、排行榜、多模式等扩展功能

## 代码风格
- 组件文件名使用 PascalCase，例如 `GameCard.tsx`
- 函数名、变量名使用 camelCase，例如 `handleSendMessage`
- 常量使用 UPPER_SNAKE_CASE，例如 `MAX_HINT_COUNT`
- 类型、接口命名使用 PascalCase，例如 `Story`, `Message`, `HintLevel`
- 自定义 Hook 以 `use` 开头，例如 `useGameSession`
- 路由页面统一放在 `pages/` 或 `app/` 目录下，组件统一放在 `components/` 目录下
- 业务枚举统一集中管理，避免在多个组件中写死字符串
- 提交前保证通过 lint 与基础类型检查

## 设计要求
- 整体风格：神秘、悬疑、略带诡异感，主色调为深蓝、深紫、黑色
- 视觉关键词：迷雾、微光、渐变、阴影、轻微不安感、仪式感
- 按钮风格：可使用不规则边缘、裂纹感、紫色或冷蓝色光晕
- 页面基调需要兼顾“可读性”和“沉浸感”，不能为了风格牺牲信息识别
- 游戏大厅建议使用卡片布局，支持故事标题、难度、题材标签展示
- 游戏页面应以聊天界面为主，突出汤面、输入框、提问历史和提示入口
- 汤底揭晓页要有更强的仪式感，可采用分段揭晓、渐显动画、关键问题高亮等方式
- 响应式适配必须覆盖移动端与桌面端，移动端优先保证输入、阅读、滚动体验

## 功能要求
- 支持游戏大厅、游戏页、结果页三个核心页面
- 支持故事难度分类：简单 / 中等 / 困难 / 地狱
- 支持故事题材分类：清汤 / 红汤 / 黑汤 / 脑洞 / 反转 / 日常
- 游戏页面需要具备以下能力：
  - 展示汤面
  - 玩家输入问题
  - AI 返回“是 / 否 / 无关”
  - 展示对话历史
  - 查看汤底
  - 结束游戏
  - 再来一局
- 必须支持提示系统：
  - 轻提示：提示推理方向
  - 中提示：排除错误路径
  - 强提示：给出关键突破口
- 必须支持复盘系统：
  - 展示玩家总提问数
  - 展示关键提问
  - 展示最终汤底
  - 给出简单推理评价
- 必须支持新手引导：
  - 给出示例问题
  - 解释什么是“是非提问”
  - 说明“无关”的含义
- 必须定义通关逻辑：
  - 用户命中核心真相时判定通关
  - 或在结果页显示“已接近正确答案，但未完整命中”

## 数据与状态要求
- 故事数据至少包含：
  - `id`
  - `title`
  - `difficulty`
  - `category`
  - `surface`
  - `bottom`
  - `keyPoints`
  - `hints`
  - `contentWarning`
- 对话消息至少包含：
  - `id`
  - `role`
  - `content`
  - `createdAt`
  - `answerType`
- 游戏会话状态建议包含：
  - 当前故事
  - 对话历史
  - 已使用提示次数
  - 当前游戏状态（进行中 / 已通关 / 已放弃 / 已结束）
  - 是否显示汤底
  - 是否为新手模式
- 建议将故事题库与游戏会话分离，避免状态混乱
- 后期如接入登录系统，需预留记录用户游戏历史的结构

## AI 行为规范
- AI 默认身份为“海龟汤主持人”
- AI 对普通事实问题只能回答：
  - “是”
  - “否”
  - “无关”
- AI 不得主动泄露汤底
- AI 不得编造与汤底冲突的新设定
- 当用户请求提示时，可根据提示等级返回引导内容
- 当用户请求总结时，只允许总结已确认信息，不得提前剧透
- 当用户提问不符合“是非问题”格式时，提醒其换一种问法
- 模型输出必须经过后处理校验，避免出现解释性长文本污染玩法体验

## 技术建议
- 前端：React + TypeScript + Vite
- 样式：Tailwind CSS
- 状态管理：优先 React Context + Hooks，复杂后可接 Zustand
- 路由：React Router
- 后端：Node.js + Express / NestJS
- AI 接入：DeepSeek / Claude / 其他支持稳定对话的模型
- 部署：
  - 前端可部署在 Vercel
  - 后端建议部署在支持环境变量和 API 转发的平台
- 推荐采用“规则 + 模型”混合模式：
  - 题库中预设汤底、关键点、提示、标签
  - AI 负责理解用户问题
  - 服务端根据规则映射后输出“是 / 否 / 无关 / 提示”
- 不建议完全依赖模型自由发挥，否则容易出现前后矛盾和逻辑漏洞

## 注意事项
- 优先实现核心闭环：选题 → 提问 → 回答 → 推理 → 揭晓 → 再来一局
- 不要一开始就投入过多时间在排行榜、社交分享、多人对战等非 MVP 功能上
- AI API Key 不要硬编码，必须使用环境变量管理
- 必须考虑内容分级，允许用户避开血腥、惊悚、死亡等敏感题材
- 必须考虑 AI 异常场景：
  - 超时
  - 返回空内容
  - 返回非法文本
  - 多轮回答矛盾
- 对于查看汤底、结束游戏等高风险操作，建议增加确认机制
- 对话列表较长时要处理滚动定位与性能问题
- 动画和氛围效果应服务于体验，不要影响输入、阅读和推理效率

## 测试要求
- 每个核心功能完成后进行手动测试
- 测试游戏基本流程是否完整：
  - 能否进入大厅
  - 能否开始游戏
  - 能否正常提问
  - AI 是否按规则回答
  - 能否查看汤底
  - 能否结束游戏
  - 能否再来一局
- 测试提示系统是否按级别正常工作
- 测试通关判定是否合理，避免“用户已经猜中但系统不认”
- 测试不同难度、不同题材故事是否可正常游玩
- 测试桌面端、平板端、移动端的显示效果
- 测试长文本、连续多轮提问、频繁切换页面等边界场景
- 测试 AI 返回异常时的兜底策略是否生效
- 如上线前有条件，至少补充基础单元测试与关键流程集成测试

## 后续扩展建议
- 用户登录与游戏记录保存
- 排行榜与成就系统
- 多人合作 / 对战模式
- 自定义海龟汤创建功能
- AI 主持人口吻切换（神秘 / 冷淡 / 严肃 / 戏谑）
- 更完整的复盘分析页
- 埋点系统与数据分析看板
